# ADR-0002：人工检测录入的产品价值、可行性与入口设计

**状态（Status）：** Proposed
**日期（Date）：** 2026-07-20
**决策者（Deciders）：** Ingot 维护者（@liuweichaox）
**范围：** `InspectionRecord` 提交路径的端到端设计（契约 → 通道 → 界面 → 身份）

---

## 1. 价值评估（Value）

### 1.1 Ingot 在解决什么真问题

制造业追溯的真实痛点不是"没有数据"，而是**数据分散且互不可信**：设备时序在 SCADA、工单在 MES、物料在 ERP、检测在量具软件和 Excel、异常处置在纸质单和微信群。追一个批次的问题要跨 5 个系统人工拼时间线，拼完还无法证明这条时间线是完整的。

Ingot 的定位切得很准：**不做 MES / SCADA / QMS 的替代，只做它们上面薄薄一层"可信记录层 + 可追溯问答"**。三个设计决定了它的价值上限：

| 决定 | 为什么这是对的 |
|---|---|
| 不嵌入任何设备协议、不写 PLC | 规避了工业软件最重、最难维护、最容易背锅的部分；也让安全审查可通过 |
| Chat 只调用注册过的只读数据工具，数字必须来自工具结果并带附件 | 这是工业场景 LLM 唯一可落地的形态。一次幻觉数字就会让整个系统在质量部门失去信任 |
| MIT 开源 + Docker 本地自托管 | 大量工厂数据不允许出厂区。这是对 SaaS 竞品的结构性优势 |

### 1.2 被低估的那个卖点

目前代码里最有说服力的能力不是"Chat 能回答问题"，而是 **`check_data_quality`**：周期配对、序号缺口、生产信息缺失、数据新鲜度。

这意味着 Ingot 可以在客户**还没有任何分析需求**时就交付价值——告诉他们"你以为你有数据，其实 37% 的加工周期没有配对的完工事件，最近 6 小时 EDGE-003 没有任何数据"。这是一个可以独立售卖、独立验证、无需 LLM 也成立的结论。

**建议**：把"生产数据体检"作为首屏和第一个 demo，Chat 作为体检之后的深入手段。当前 README 把 Chat 放在最前面，实际上是把最需要信任门槛的能力放在了最前面。

### 1.3 价值的致命依赖

Ingot 的价值 = 记录完整度 × 提问价值。记录不全，Chat 只会输出一串 limitations。而记录完整度里，**设备侧靠适配器（一次性工程），人工侧靠一线操作工每天的配合（永久性摩擦）**。

所以本文档第 3 节的人工录入设计，不是一个次要功能，而是**决定整个平台价值能否兑现的那一环**。

### 1.4 竞争位置（诚实评估）

- **MES 自带报表 / Grafana 自建**：便宜、已存在。Ingot 必须赢在"跨系统 + 可追溯问答"，赢不了单系统看板。
- **Seeq / Tulip / Ignition**：功能更全、生态更大、价格高、多为云或重部署。Ingot 赢在轻、开源、自托管、契约简单。
- **真正的对手是 Excel**。这是所有制造业数字化项目的真实对手，也是第 3 节的设计基准线：**比 Excel 更快，才有人用**。

## 2. 可行性评估（Feasibility）

### 2.1 技术可行性：高

`.NET 10 + PostgreSQL/TimescaleDB + Vue 3`，约 10.6k 行 C#、1.8k 行 Vue，646 次提交。分层清晰（Contracts / Domain / Application / Infrastructure / Api），契约校验、幂等（`eventId` + `(edgeId, seq)`）、SSE 断点续传（`Last-Event-ID`）、用户 隔离都已实现且有单元测试。

技术上没有未解难题。最难的部分（设备协议、实时控制）被主动排除在边界之外。

### 2.2 真正的风险是冷启动，不是技术

代码走查发现的关键缺口，**全部集中在人工侧**：

| # | 缺口 | 影响 |
|---|---|---|
| 1 | **没有附件上传端点**。`InspectionAttachment` 要求 `StorageRef` + `Sha256` + `SizeBytes`，注释写"已进入受控附件暂存区"，但仓库里不存在任何 multipart / 上传 / 对象存储实现 | 契约悬空。客户端无法自行满足，"拍照上传超差工件"这个最刚需的场景**目前不可能完成** |
| 2 | **Platform Web 没有任何检测录入界面**（仅 Chat / Edges / Events / Logs / Metrics） | 人工提交只能靠 curl。等于没有人工通道 |
| 3 | **没有检测定义（`DefinitionCode` / `CharacteristicCode`）注册表**。二者是自由字符串，上下限由提交方随手带入 | 无法生成表单、无法校验、无法保证同一检测项在不同工位口径一致。数据可信度从源头就漏了 |
| 4 | **身份模型不足**。`SubmittedBy` 即 token map 的 key，`SubmitterVerified` 只反映"是否开启了 token 校验" | 无法区分"3 号工位"和"3 号工位上的张三"。质量追溯里人员责任是硬要求 |
| 5 | **没有适配器样例 / 模板** | 第一个客户要从零写全部映射。采用摩擦最高的一环没有任何脚手架 |
| 6 | 测试集中在契约与 token 层，缺集成 / E2E | 部署到真实工厂前的信心不足 |

### 2.3 结论

**可行，但当前状态是"机器侧 API 完备、人工侧接近于零"。** 缺口 1–4 不是打磨项，是阻断项：在补齐之前，任何真实工厂试点都无法启动人工检测这条线。

## 3. 决策：人工录入怎么设计才有人愿意用

### 3.0 设计基准线

> 一线操作工不会为任何系统每件多花 30 秒。设计目标不是"功能齐全"，是**比现在填 Excel / 纸单更快**。

违反这条的方案，无论多完整，最终结果都是"工位上放一台没人碰的平板，数据靠班组长下班前补录"——那种数据比没有数据更危险，因为它看起来是真的。

### 3.1 定义驱动：消除一切自由输入

新增 `InspectionDefinition` 注册表：

```text
InspectionDefinition
├─ definitionCode / version          （不可变，版本化）
├─ appliesTo: 工序 / 工位 / 产品族
└─ characteristics[]
   ├─ characteristicCode / 显示名
   ├─ inputType: numeric | enum | boolean | photo
   ├─ unit（UCUM）
   ├─ lowerLimit / upperLimit / target   ← 上下限的唯一权威来源
   └─ required
```

**表单由定义生成，不由前端硬编码。** 操作工只填数值，`definitionCode` / `unit` / `上下限` / `characteristicCode` 全部由服务端预填。

这一步同时解决三件事：录入速度、数据一致性、以及**上下限不再由提交方自证**（当前设计里提交方可以自己声明限值，这在质量审计上站不住）。

### 3.2 入口：扫码即关联信息，零导航

```text
扫工件二维码 / 工单条码
  └─ GET /api/v1/inspection-context?workpieceId=WP-001
       ├─ 解析出 workpieceId、operationRunId（来自已有 ProductionEvent 记录）
       ├─ 返回该工位当前应做的 InspectionDefinition 列表
       └─ 返回已提交项（避免重复录入）
```

操作工的完整动作序列：**扫码 → 数字键盘输入 → 提交**。没有下拉框、没有搜索、没有"请选择工单"。

`operationRunId` 从事件记录反查而不是让人填，是这个设计里最省事的一刀——它同时消灭了最容易填错、且填错就无法关联的那个字段。

### 3.3 三条通道并存，不要只做一条

| 通道 | 形态 | 覆盖场景 | 优先级 |
|---|---|---|---|
| **A 工位快捷录入** | 移动端 PWA，扫码 + 大号数字键盘 | 日常单件在线检测 | P0 |
| **B 批量导入** | 粘贴 / CSV 上传 → 列映射 → 预校验 → 批量提交 | 量具软件导出、存量补录、班末补传 | P1 |
| **C 仪器直连** | 三坐标 / 卡尺 / 试验机直接调 API | 逐步替代 A | P2 |

常见错误是只做 A。**B 是采用率的关键**：质检员手上永远已经有一份 Excel，不给导入通道，他们就不会迁移，只会两头都不做。B 也是唯一能把历史数据灌进来的路径，而没有历史数据的生产记录库讲不出趋势。

### 3.4 离线优先（车间 Wi-Fi 一定会断）

契约层已经准备好了：`RecordId` 是客户端生成的 UUIDv7（幂等），`RecordedAt` 与 `MeasuredAt` 分离（支持补传）。**前端必须兑现这个设计**：

```text
提交 → 写入 IndexedDB 本地队列 → 立即返回"已保存"
        └─ 后台重试上传 → 成功后标记已同步
```

操作工永远看到"已保存"，不会看到"网络错误请重试"。断网 4 小时的产线，恢复后自动补齐。

### 3.5 补齐附件上传通道（阻断项）

```text
POST /api/v1/inspection-attachments   (multipart/form-data)
  → 服务端计算 sha256、落地受控存储、返回
    { attachmentsId(UUIDv7), storageRef, sha256, mediaType, fileName, sizeBytes }
  → 客户端把返回体原样放进 InspectionRecord.attachments[]
```

关键点：**sha256 由服务端算，不由客户端算**。当前契约要求客户端提供 hash，浏览器端可做但徒增复杂度，且客户端自证 hash 在防篡改含义上本来就没有意义。

有了这个端点，"超差 → 拍照 → 一起提交"才成立，而这恰恰是人工检测最有价值的那部分记录。

### 3.6 让操作工成为受益者，而不只是数据来源（最关键的一节）

**只索取不回馈的系统，一定会被绕过。** 提交成功后立即返回并展示：

- 本次结果相对上下限的位置（可视化，不只是 PASS/FAIL 文字）
- 该特性最近 20 次的趋势 —— "你这台机器今天在往上漂"
- 本班次该工位的合格率

这让录入从"给上面交差"变成"我自己要看的东西"。这是所有工位数据采集项目里，采用率高与低的分水岭，比任何 UI 打磨都重要。

### 3.7 身份：工位是 token，人是 PIN

```text
Authorization: Bearer <工位 token>      ← 长期，配置在设备上
X-Operator: <工牌扫码 / 4 位 PIN>       ← 每班一次，非每次提交

SubmittedBy       = 人员 ID
SubmitterVerified = 工位 token 有效 && 人员身份已核验
```

当前 `SubmittedBy` 直接等于 token key，导致"人"和"设备"混为一谈，`SubmitterVerified` 也就失去了信息量。分离之后它才真正可用于质量追溯。

**注意**：每班一次登录，不是每次提交登录。后者会直接杀死 3.0 的基准线。

### 3.8 明确的反模式

| 不要做 | 原因 |
|---|---|
| 必填备注 / 原因 | 结果是全场"1"、"无"、"ok"，污染数据还浪费时间 |
| 提交确认弹窗 | 每天几百次，第三天起就成为肌肉记忆盲点，不再起校验作用 |
| 超差时阻断提交 | Ingot 记录**记录**，不做处置。阻断会诱导操作工改数据——这是最坏的结果 |
| 每次提交都要登录 | 见 3.7 |
| 复用 Element Plus 桌面表单直接塞进移动端 | 工位场景需要大按钮、数字键盘、单手可达、戴手套可点 |

### 3.9 超差的正确处理

不阻断，但要**响亮**：本地立即红色高亮 + 记录为 `FAIL` + 触发 webhook 通知质量。记录照录，处置权交给 QMS —— 这与 README 里"不替代质量处置系统"的定位保持一致。

## 4. 落地顺序

| 阶段 | 内容 | 解除的阻断 |
|---|---|---|
| **P0** | `POST /api/v1/inspection-attachments` 上传端点<br>`InspectionDefinition` 注册表 + CRUD<br>Platform Web 检测录入页（桌面版即可） | 缺口 1、2、3 —— 之前无法启动任何人工试点 |
| **P1** | 扫码取关联信息（`inspection-context`）<br>移动端 PWA + IndexedDB 离线队列<br>提交后即时反馈（3.6） | 采用率 |
| **P2** | CSV / 粘贴批量导入 + 列映射<br>工位 token 与人员身份分离 | 存量数据迁移、质量追溯合规 |
| **P3** | 参考适配器模板（Modbus / OPC UA / MES 数据库轮询各一个样例） | 冷启动摩擦（2.2 缺口 5） |

## 5. 权衡与代价（Trade-offs）

| 决策 | 收益 | 代价 |
|---|---|---|
| 引入 `InspectionDefinition` 注册表 | 数据一致、表单自动生成、限值有权威来源 | Ingot 从"纯记录层"向"轻主数据管理"越界一步；需要定义的维护界面和版本迁移策略 |
| 离线优先前端 | 车间可用性 | 前端复杂度显著上升；本地队列的冲突与过期需要策略 |
| 服务端保管附件 | 上传通道成立、hash 可信 | Platform 从此有了对象存储依赖和容量/保留策略问题 |
| 移动端 PWA 独立于 Platform Web | 工位体验达标 | 多一个前端工程要维护；建议共用 API 层与类型定义 |
| 不阻断超差提交 | 数据真实、定位清晰 | 需要 webhook + 质量侧流程配合，否则超差记录无人跟进 |

## 6. 需要复审的时点

- **接入第 3 个客户时**：`InspectionDefinition` 是否需要支持继承 / 产品族覆盖。
- **单日人工提交超过 1 万条时**：`inspection_records` 是否也要进 hypertable（当前只有 `production_events` 是）。
- **出现第一次质量审计需求时**：`SubmitterVerified` 与附件链是否满足客户所在行业的合规口径（IATF 16949 / GMP 差异很大）。
- **附件存储超过 1 TB 时**：受控存储的分层与保留策略。
