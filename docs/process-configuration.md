# 工艺配置模型

Ingot 将工艺配置拆分成三个独立、版本化的主体，避免把数据定义、运行值和分析选择混在同一份 Profile 中。

## 工艺数据模型

工艺数据模型描述原始数据如何被平台理解，包括：

- 采集数据项：来源字段、标准数据项编码、数据类型、单位和数据类别；
- 配方参数定义：允许出现的参数编码、来源字段、类型、单位和可空性；
- 工艺阶段：控制器步序、稳定阶段编码、显示名称和预计时长。

工艺数据模型不保存配方实际值，也不决定哪些信号参与某一种分析。已发布版本不可原地修改。

API：`/api/v1/process-data-models`

## 配方版本

配方版本必须引用一个工艺数据模型版本，并保存该版本的完整有效参数值。`basedOnVersion` 记录沿用关系；Ingot 不要求记录参数修改原因。

适用条件使用通用上下文键值，例如 `product.series=LENS-A`。生产开始时，生产条件快照冻结实际使用的配方编号、版本和有效参数，历史查询不再动态继承当前配方。

API：`/api/v1/recipe-versions`

## 分析方案

分析方案引用工艺数据模型，独立定义：

- 分析范围：生产周期、生产运行段或分析窗口；
- 对齐方式：按工艺阶段、开始后时间或完成度归一化；
- 生产条件筛选；
- 同类比较键，例如产品系列、材料牌号或工装组合；
- 分组维度，例如 `quality.outcome`；
- 参与分析的数据项、是否保留完整曲线以及派生特征。

质量结果是独立的一等数据。分析方案只引用质量分组维度，不在方案中复制或降级保存质检证据。

空的生产条件筛选只表示“适用于所引用的工艺数据模型”，不会跨数据模型或跨行业匹配。运行时先从事件固化的 `data_model_id/data_model_version` 或配方版本解析数据模型，再选择同模型内的已发布分析方案。配置的同类比较键必须在每个候选上下文中有值；缺失时拒绝比较，不能把两个空值当成同类。

生产周期比较读取完整周期事件；连续运行设备使用生产运行段或用户定义时间窗口。接口可内部分页，但分页大小不是分析上限。

API：`/api/v1/process-analysis-plans`

## 分析展示

Platform Web 使用 Plotly 作为统一图表渲染层，但图表配置不是新的业务数据主体。分析方案决定分析对象、比较条件、对齐方式和数据项；页面再根据结果选择趋势、分布、质量分层或统计明细视图。

- 周期和时间窗口趋势使用统一相对时间轴，基准对象保持实线并突出显示；
- 页面通过游标读取完整采样轨迹，不把单页 500 条当作数据分析上限；
- 质量分层图、统计卡和明细表共享相同筛选范围，图表用于识别差异，表格用于复核数值和追溯记录；
- 图表支持悬浮联读、缩放、复位、响应式布局和导出图片；AI 返回的结构化图表也使用同一个渲染组件，并保留可展开的数据表。

## 生命周期

三个主体都支持 `draft`、`published` 和 `retired`。草稿可以修改或删除；已发布版本只能停用，其他变化必须创建新版本。

旧入口 `/profiles` 重定向到 `/configuration/process-data-models`。
